Arrays and Hashes

February 22nd, 2015

A big week for us this week in DevBootcamp -- we started learning the basics for the first time of Ruby, the main complicated programming language that we'll be learning over our 18 weeks here. In general, most of the things we're learning in DevBootcamp will be the tools for making an app work right, once you have the app in place -- HTML and CSS to make it look right, JavaScript to add interactive elements, AJAX to make those interactive elements visually impressive, Git to get it all loaded to the web and regularly maintained -- but Ruby is what we use to actually program the app itself, so if anything could be considered the "main" point of going through DevBootcamp, this would be it. And we've even started creating our own standalone programs from scratch too, here in our very first week, although certainly it's no next Facebook at this point...

There's all kinds of basic Ruby things we've been studying up on this week -- objects, variables, methods, operators, a refresher course on basic math symbols (which embarrassingly I needed a lot more than I realized I would); but the thing DBC has asked us to concentrate on in this week's technical blog post are the related subjects of arrays and hashes. And like everything else in Ruby, this is both a simple subject and a complicated one; because when all is said and done, an array is simply a series of objects, whether those are integers you're talking about (or that is, numbers that Ruby recognizes as numbers), or strings (defined as a series of characters that have been put next to each other, kind of like the letters in a word; numbers can be strings as well, but in that case, Ruby only recognizes them as English-language characters like words, and not as actual numbers that can be added and subtracted together), in all cases defined by putting a series of brackets around the entire set. So here's an array of numbers, for example...

1
x = [1, 4, 7, 11, 15, 21]

While here's the same array but expressed as strings...

1
x = ["1", "4", "7", "11", "15", "21"]

But where it gets complicated, of course, is that you can do all kinds of things to these arrays, which is where the real power of Ruby lies but is also where it's very easy to get confused. And this is just the start of the rabbithole that one tumbles down when starting to get involved with Ruby and arrays, but this should at least give you non-techies an idea of what programmers are talking about when they bring up this subject at a dinner party.

So what's the difference between an array and a hash? Well, not a lot when all is said and done; but to understand that small difference, you need to understand the way Ruby treats the items in an array so that it can call on each individual item when needed. And that's basically by assigning a number to each item as it goes down the list, but with the big catch that such number lists actually start with 0 in Ruby, not 1. So if you defined the following array as a series of strings, for example...

1
sentence = ["The", "quick", "brown", "fox"]

...then Ruby would consider the object "sentence[0]" to be "The", the object "sentence[1]" to be "quick", the object "sentence[2]" to be "brown", etc, so that if you want to call on any of these objects in the future, you have to memorize which numbers go with which items. Hashes basically serve the same function, but allows you as the programmer to assign whatever label (or "key") to them you want, like so...

sentence = {
   sentencefirst => The,
   sentencesecond => quick,
   sentencethird => brown,
   sentencefourth => fox
}

Granted, it doesn't make much sense why you would do it that way with this particular example, but think instead about putting together a ticket for a rock concert, and defining all the data associated with that ticket...

So as you can see, the combination of array manipulation plus the customization of hashes can be a hugely powerful thing within a computer program, and is the key towards being able to manipulate the large databases of information that are usually the backbone of most computer apps. It all of course gets much, much more complicated than this; but this hopefully gives you at least a basic idea of what a programmer is talking about when they start talking about the trouble they were having with their hashes at work last week.
