DevBootcamp Technical Blog Week 5

Classes

March 7, 2015

I've talked here in previous weeks about how one of the things Ruby is best known for is being a "plain language" programming language, and that the process of putting together a good Ruby program can in many ways be compared to the process of putting together a complete English-language sentence, with all the various parts that such a sentence needs in order to be considered "complete." So if variables are nouns in such a metaphor, and methods are verbs, then you might consider classes to be the "table of contents;" although this is where the metaphor starts breaking down a little, because a class is not just a top-level look at everything that's going on in that program, but is a chance to define one of those top-level things, in a kind of sophisticated and highly detailed way that makes Ruby so powerful, so that a programmer can keep calling on that definition over and over without having to redefine it all each time they do.

What do I mean by that? Well, perhaps it'd be best to use an example from a really great and informative book we're currently reading as part of our homework in DevBootcamp, Sandi Metz's Practical Object-Oriented Design in Ruby. She talks in an early chapter there about how, say, the owner of a bicycle store might need to keep track of a series of details about various bikes they sell, so that they'll know what spare parts to bring with them when conducting a guided long tour for their customers. So that might involve, for example, defining the size of the chain ring on that bike, and how many cog teeth there are on a given gear...

A class, then, simply gathers up this information all in one spot, so that it can be used over and over again...

class Bike
  attr_reader :chainring, :cog
  def initialize(chainring, cog)
    @chainring = chainring
    @cog = cog
  end

  def ratio
    chainring / cog.to_f
  end
end

And then every time you want to add a new type of bike, each of which will have different values for these variables, you create a new instance of the "Bike" class, a process known in programming as "instantiation..."

mountain_bike = Bike.new(30, 27)

Now, as always with these non-tech blog entries of mine, the subject is actually a lot more complicated than this, or you would never need to go to a programming class to learn this stuff in the first place; for a good example, I'm not bothering today to explain the various details you're seeing in this code above, like what an "attr_reader" is or why it's so important to have an "initialize" method in your class to begin with. (And that's because I'm having an extra-hard time this week with our assigned challenges, so I want to get back to them right away and spend just a minimum amount of time this week writing out these blog posts.) But what you're seeing above is a class explained in about the most basic way possible; and this is the real power of an object-oriented programming language, that you can define these big complicated things for yourself and then call on them over and over, essentially making the programming language do whatever it is you have in mind, no matter what in particular that is.

Okay, a short and sweet blog entry for today, but I hope you'll have a chance to come by again next week for more!
